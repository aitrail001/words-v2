# Lexicon Tool Design

**Status:** APPROVED  
**Date:** 2026-03-07  
**Scope:** Offline/admin lexicon generation pipeline for WordNet-backed vocabulary, learner-facing LLM enrichment, validation, snapshot packaging, and local DB import.  
**Live Status Board:** `docs/status/project-status.md`

---

## Goal

Create a separate lexicon-generation tool inside this repository that:

1. uses WordNet as the canonical lexical backbone
2. uses `wordfreq` as the frequency reference
3. uses an LLM to generate learner-facing enrichment fields
4. supports future non-WordNet entries such as idioms, phrasal verbs, and curated missing words
5. produces versioned snapshot outputs that can later populate the local application database

The tool is an offline/admin workflow. It is not part of the normal learner-facing request path.

---

## Product Fit

This project teaches English at the meaning level and plans to support concept-first learning via WordNet synsets. That makes a hybrid lexicon pipeline a strong fit:

- **WordNet** gives lexical identity, sense boundaries, and concept structure.
- **wordfreq** gives practical ordering and coverage prioritization.
- **LLM enrichment** adds learner-facing value that WordNet alone does not provide well: clearer definitions, sense-linked examples, collocations, confusable words, usage notes, and register/domain framing.
- **custom expression support** is needed because WordNet will not cover all learner-important items.

This tool should therefore become the lexicon content pipeline behind the product, while leaving runtime lookup/import/review behavior to the main app.

---

## V1 Decision Summary

### Chosen approach

Use a **Python CLI tool with normalized internal snapshot files plus a compiled export JSONL**.

### Approved next-slice decisions

- keep `build-base` and `enrich` as separate offline stages
- make `build-base` use real `WordNet` + `wordfreq` providers
- make `build-base` fail loudly when lexical dependencies are unavailable instead of silently falling back
- add a dedicated `enrich` command that reads normalized snapshot files and writes `enrichments.jsonl`
- keep DB import separate from enrichment so operators can review, validate, compile, and import independently

### Why this approach

- better auditability than one giant generated file
- easier regeneration of only one layer
- clearer provenance and validation boundaries
- flexible enough for future idioms, custom expressions, and human overrides
- still simple to import, because the final compiled export can match the learner-facing JSONL schema exactly

### Not in v1

- no dedicated admin web interface for generation
- no live generation from the app runtime
- no automatic import immediately after generation

An admin UI may be added later for review, approval, diffing, and import control.

## Proposed Location

Put the tool under a project-local Python package and CLI entrypoint:

- `tools/lexicon/` — tool package and CLI modules
- `tools/lexicon/tests/` — unit/integration tests for the tool
- `data/lexicon/` — ignored local output area for snapshots

This keeps the tool separate from learner runtime concerns, while still living inside the repo and sharing Python infrastructure when needed.

## High-Level Architecture

Pipeline stages:

```text
seed config / word selection
-> build WordNet-backed lexical base
-> attach wordfreq ranking
-> generate learner-facing enrichment via LLM
-> validate structure + semantics
-> package versioned snapshot
-> compile final export JSONL
-> import chosen snapshot into local DB
```

Key principles:

- **canonical lexical identity is not generated by the LLM**
- **learner-facing text is generated or curated as a separate enrichment layer**
- **operators must be able to rerun enrichment without regenerating the lexical base**
- **base generation should stop clearly if required lexical data providers are missing**

## Output Model

The tool should emit two layers of output.

### 1. Normalized snapshot files

These are internal pipeline artifacts. They are linked by stable IDs.

Recommended files:

- `manifest.json`
- `lexemes.jsonl`
- `senses.jsonl`
- `concepts.jsonl`
- `enrichments.jsonl`
- `expressions.jsonl`
- `validation_report.csv`
- `errors.jsonl`

### 2. Compiled final export

This is the import-friendly flattened file:

- `words.enriched.jsonl`

This compiled file should contain the full learner-facing JSONL schema fields:

1. `schema_version`
2. `word`
3. `part_of_speech`
4. `cefr_level`
5. `frequency_rank`
6. `forms`
7. `senses`
8. `confusable_words`
9. `generated_at`

So the internal model is normalized, but the downstream export still preserves the complete single-record-per-word schema.

---

## Linking Strategy Across Files

Internal files must link by stable IDs, never by raw word text alone.

### Core IDs

- `snapshot_id` — identifies one generated snapshot
- `lexeme_id` — stable lexical headword record
- `sense_id` — one learner-visible sense tied to a lexeme
- `concept_id` — one concept/synset-level grouping
- `enrichment_id` — one generated enrichment version for a sense
- `expression_id` — one idiom/phrasal/custom expression entry

### Canonical anchors

Where available, store:

- `wn_synset_id`
- `wn_lemma_key` or equivalent WordNet-specific identifier
- source/provenance metadata

### Why

This enables:

- multiple senses per word without ambiguity
- multiple enrichment versions for the same sense
- support for custom entries without breaking canonical WordNet-backed entries
- clean regeneration and diffing

## Recommended Internal Record Shapes

### `lexemes.jsonl`

Purpose: one row per lemma/headword.

Recommended fields:

- `snapshot_id`
- `lexeme_id`
- `lemma`
- `language`
- `wordfreq_rank`
- `is_wordnet_backed`
- `source_refs`
- `created_at`

### `senses.jsonl`

Purpose: one row per selected learner-visible sense.

Recommended fields:

- `snapshot_id`
- `sense_id`
- `lexeme_id`
- `wn_synset_id` nullable
- `part_of_speech`
- `canonical_gloss`
- `selection_reason`
- `sense_order`
- `is_high_polysemy`
- `created_at`

### `concepts.jsonl`

Purpose: concept/synset grouping for concept-first learning.

Recommended fields:

- `snapshot_id`
- `concept_id`
- `wn_synset_id` nullable
- `canonical_label`
- `part_of_speech`
- `gloss`
- `lemma_ids`
- `created_at`

### `enrichments.jsonl`

Purpose: learner-facing generated content per sense.

Recommended fields:

- `snapshot_id`
- `enrichment_id`
- `sense_id`
- `definition`
- `examples`
- `cefr_level`
- `primary_domain`
- `secondary_domains`
- `register`
- `synonyms`
- `antonyms`
- `collocations`
- `grammar_patterns`
- `usage_note`
- `forms`
- `confusable_words`
- `model_name`
- `prompt_version`
- `generation_run_id`
- `confidence`
- `review_status`
- `generated_at`

### `expressions.jsonl`

Purpose: non-WordNet or custom entries such as idioms, phrasal verbs, and missing vocabulary.

Recommended fields:

- `snapshot_id`
- `expression_id`
- `expression_text`
- `expression_type` (`idiom`, `phrasal_verb`, `custom_word`, `fixed_phrase`)
- `linked_lexeme_ids`
- `linked_concept_ids`
- `base_definition`
- `enrichment_ref`
- `source_type` (`custom`, `curated`, `llm`, `dictionary_api`)
- `source_reference`
- `created_at`

## Operator Flow

The approved V1 operator flow is:

```text
build-base --output-dir <snapshot_dir>
-> enrich --snapshot-dir <snapshot_dir>
-> validate --snapshot-dir <snapshot_dir>
-> compile-export --snapshot-dir <snapshot_dir> --output <compiled_jsonl>
-> import-db --input <compiled_jsonl> [--dry-run or apply]
```

This split keeps lexical base generation, LLM enrichment, validation, compilation, and DB import independently rerunnable.

---

## Dependency Policy

`build-base` depends on `WordNet` and `wordfreq` being present and usable.

Approved behavior:

- if `WordNet` data is unavailable, `build-base` exits with a clear error
- if `wordfreq` is unavailable, `build-base` exits with a clear error
- bootstrap fake providers are allowed in unit tests, but not as silent runtime fallback for the operator path

`enrich` should use environment-driven provider configuration only. No secrets are hardcoded in the repo or written to snapshot outputs.
